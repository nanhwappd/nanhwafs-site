# 南华官网建设与维护指南

**版本 v1.0 · 2026-07-12 · 适用对象：零编程基础的老师 / 未来接手的维护员 / 协助施工的 AI 助手**

> 这份指南的目标：让一个从没碰过代码的人，照着做也能安全地改网站、发新闻、不弄坏东西。
> 看不懂的地方，把这份指南整份丢给 AI 助手（Claude / ChatGPT 等），说「照这份指南帮我做 XX」。

---

## 0. 这个网站是什么（5 分钟看懂）

南华官网是一个**静态网站**——意思是：它就是一堆写好的网页文件（.html），没有后台、没有数据库、没有登录系统。这带来两个好处：

1. **永远不会被黑、不会崩**：没有后台就没有可入侵的东西。
2. **完全免费**：托管在 GitHub Pages 上，零服务器费用。

整个系统只有三个地方，认清它们就懂了一半：

| 地方 | 是什么 | 在哪 |
|---|---|---|
| **本机工作目录** | 网站的「源文件」，你改动的地方 | `C:\dev\nanhwafs-site` |
| **GitHub 仓库** | 源文件的云端备份 + 历史记录（每次改动都留底，永远可反悔） | github.com 上的 `nanhwappd/nanhwafs-site` |
| **线上网站** | 家长实际看到的网站，GitHub 自动从仓库生成 | `https://nanhwappd.github.io/nanhwafs-site`（P5 后换正式域名） |

**改网站的完整流程永远是这四步**（细节见第 4 章）：

```
① 改本机文件 → ② 本地预览确认 → ③ 提交(commit) → ④ 推送(push)
                                                    ↓
                                     1~2 分钟后线上网站自动更新
```

只要没做第 ④ 步，线上网站就纹丝不动——所以在本机怎么试验都是安全的。

## 2. 六条铁律（改任何东西之前先过一遍）

这些是建站时已经反复论证、拍板定案的规矩。**不理解没关系，遵守就行**；想推翻任何一条，先找到当初的理由再说。

1. **受众是「正在择校的家长」**，不是校史研究者。每个页面自问：一个考虑要不要送孩子来的家长，看到这页会更想来吗？
2. **照片优先（photo-first）**：有 8 年真实照片库，任何页面输出大段文字墙 = 不合格。但照片之间要有叙事线，不是堆图。照片第一来源永远是 FB 档案真实照片，**不用 AI 生成图、不用网络图库**。
3. **手机优先**：多数家长用手机看。任何改动先保证手机（375px 窄屏）显示正常，再看电脑。验收时两种都要看。
4. **全站简体中文**。历史繁体内容已由脚本统一转换，新内容直接写简体。
5. **大文件不进仓库**：视频上传学校 YouTube 频道后内嵌（找冠铭）；6GB 照片母本放 `D:\Fb_AllRecord` + Google Drive 镜像，仓库里只放压缩过的精选图（长边 ≤1280px，每张约 150–250KB）。
6. **新闻页「骨架与皮肤分离」**：要改 1198 篇新闻页的外观，只改 `assets/news.css` 这一个文件；永远不要批量改新闻页的 HTML 本体。

## 3. 新维护员的一次性准备（装一次，用到底）

在新电脑上接手，需要装四样免费软件（每样都是官网下载、一路下一步）：

1. **Git**（git-scm.com）——文件版本管理
2. **GitHub CLI**（cli.github.com）——命令 `gh`，管登录授权
3. **Python 3**（python.org，安装时勾选 "Add to PATH"）——跑施工脚本
4. **VS Code**（code.visualstudio.com）——改文件用的编辑器

然后打开「终端」（开始菜单搜 PowerShell），逐行执行：

```
gh auth login                          # 登录 GitHub，用 nanhwappd 账号，照提示走
gh auth setup-git                      # 让 git 用这个账号的权限（重要！不做这步 push 会报 403）
git clone https://github.com/nanhwappd/nanhwafs-site C:\dev\nanhwafs-site
pip install pillow opencc              # 施工脚本的两个依赖
```

装完就永久可用。账号密码等交接信息见第 6 章。

## 4. 日常操作食谱（照抄即可）

以下命令都在 PowerShell 里执行。先进入网站目录：

```
cd C:\dev\nanhwafs-site
```

### 4.1 本地预览（改任何东西前后都先看一眼）

```
python -m http.server 8410
```

然后浏览器打开 `http://localhost:8410`。看完回终端按 `Ctrl+C` 关掉。
（如果报「端口被占用」，说明上次没关，直接用浏览器打开那个地址就行。）

### 4.2 改错字 / 改一段文字

1. 用 VS Code 打开要改的 .html 文件（不确定是哪个文件？在 VS Code 里按 `Ctrl+Shift+F` 全局搜索那段文字）。
2. 直接改文字，保存。
3. 本地预览确认（4.1），手机宽度也看一下（浏览器按 F12 → 点手机图标）。
4. 上线（4.5）。

### 4.3 发一篇新新闻（复制模板法，最安全）

1. 进 `news\2026\`（当年的文件夹），复制任何一篇现成的新闻页，重命名为 `日期-序号.html`，如 `20260901-1.html`（当天第 2 篇就是 `20260901-2.html`）。
2. 用 VS Code 打开，替换：标题、日期、正文、图片。
3. 图片先压缩（见 4.4），放进 `news\2026\media\`，HTML 里引用相对路径。
4. 打开 `news\2026\index.html`（年度索引页），仿照现有条目，在最上方加一条指向新页面的链接。
5. 本地预览 → 上线。

> 发完官网后，再去脸书贴摘要 + 附官网链接（官网首发、FB 转发，这是定好的顺序，对搜索引擎收录有利）。

### 4.4 加 / 换照片

铁律五：进仓库的图必须压缩过。最省事的办法是交给 AI：「帮我把这几张图压到长边 1280px、质量 78」。自己动手可用任何压图工具（如 squoosh.app 网页版），目标：**长边不超过 1280px，单张 150–250KB**。

首页 Hero 级大图是唯一例外——用原生高清图，不用压缩版放大充数。

### 4.5 上线三步（把改动推到线上网站）

```
git add -A
git commit -m "一句话写清楚这次改了什么"
git push
```

等 1–2 分钟，刷新线上网站（`Ctrl+F5` 强制刷新）即可看到。

> **push 报错 403？** 执行 `gh auth setup-git` 再 push（第 3 章漏做的那步）。

### 4.6 每季度：脸书档案增量补档（建议交给 AI 做）

新闻馆的 1198 页不是手写的，是两支脚本从脸书导出档生成的：

```
脸书导出的 HTML 档 → scripts/parse_fb_export.py → data/posts.json → scripts/build_news.py → news/ 全部页面
```

- `build_news.py` 直接运行：`python scripts\build_news.py`。它是**幂等**的——重复跑不会弄坏东西，已生成的会跳过。
- 它需要照片母本盘 `D:\Fb_AllRecord` 在位才能取图。
- 每季度流程：FB 后台导出新档 → 放到母本盘 → 让 AI 照本节跑一遍两支脚本 → 抽查新生成的页面 → 上线。

## 5. 出错了怎么办

| 症状 | 原因 | 解法 |
|---|---|---|
| `git push` 报 403 | git 没用对账号 | `gh auth setup-git`，再 push |
| push 成功但网站没变 | 部署要 1–2 分钟 + 浏览器缓存 | 等 2 分钟，`Ctrl+F5` 强刷；还不行去 GitHub 仓库页看 Actions 有没有报错 |
| 本地预览打不开 | 服务没启动或端口占用 | 重跑 `python -m http.server 8410` |
| 改坏了还没上线，想反悔 | — | `git checkout -- 文件名`（还原单个文件）或 `git checkout .`（还原全部未提交改动）|
| 改坏了已经上线 | — | 别慌，历史都在。让 AI 执行「回退上一个 commit」，或自己跑 `git revert HEAD` 然后 push |
| 图片在网页上显示不出来 | 路径写错 / 文件名大小写不符 | 检查 HTML 里的路径和实际文件名是否一字不差 |
| 脚本报错 `No module named PIL/opencc` | 依赖没装 | `pip install pillow opencc` |

**万能保底原则**：只要还没 `git push`，线上永远是安全的；就算 push 了，git 历史永远可以回退。这个系统里不存在「彻底毁掉」这回事。

## 7. 让 AI 当你的施工队（推荐工作方式）

这个网站本来就是人+AI 协作建成的，维护也建议这样干：

1. **开工先喂上下文**：把本指南和 `PLAN.md` 给 AI 读，说「你先读这两份，遵守里面的铁律」。
2. **提需求说目的，不说手段**：说「帮我在师资页加一位新老师，姓名 XX，照片在这」，而不是猜要改哪个文件。
3. **验收三件套**（AI 说完成后，你亲自做）：
   - 本地预览，电脑宽度看一遍；
   - 手机宽度看一遍（F12 → 手机图标，或直接手机开线上网址）；
   - 确认没违反第 2 章铁律（尤其：有没有变成文字墙、有没有塞了 AI 生成图）。
4. **上线由你亲手执行 4.5 的三条命令**——把「按下发布键」留在人类手里，是最简单有效的安全阀。
